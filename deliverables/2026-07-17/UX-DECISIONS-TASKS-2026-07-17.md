## 一、5 个决策点：推荐方案（已定稿 ✅）

### D1 — 自动建 Task：静默生成 vs 生成后请用户确认
**✅ 定稿：生成后请用户确认（推荐）**
- **理由**：零容错用户最怕"漏建 / 错建"。静默生成一旦误判关键词（如"讨论是否签署"），用户无感知，合规动作缺失且无任何痕迹。
- **方案**：纪要保存时服务端扫描正文 → 识别**硬动作**（签署 / 盖章 / 签字）生成**候选** SignTask → 弹窗列出"已识别 N 个签署动作" → 用户编辑负责人 / 截止日 → **确认后才落库**。
- **防重**：同会议已存在 SignTask 则不重复建（`meeting.signTasks[]` 去重），避免每次保存都新建。

### D2 — B/C 直接签署：另存副本 vs 覆盖原文件
**✅ 定稿：另存副本（推荐）**
- **理由**：不可逆覆盖是零容错红线。另存副本保留原文件可追溯，新文件带后缀，原文件仍可用于其他流程（如法务复核）。
- **方案**：完成签署时**新增**一条 Document（`suffix='signed' | 'ctc'`），原文件 `stage` 不变，两者并存。

### D3 — 归档锁定：硬锁死 vs 可解除 + 审计
**✅ 定稿：可解除 + 审计日志（推荐）**
- **理由**：合规既要"锁"（防误改）也要"留后路"（真实业务常需补一份扫描件）。硬锁死会让真实业务卡死、反而逼用户绕开系统。
- **方案**：文档加 `archivedAt` + `unlockBy / unlockAt / unlockReason` 字段；**解除归档**必须写 `AuditLog`（谁 / 何时 / 为何）。

### D4 — 文件分类体系：枚举来源
**✅ 定稿：产品定义最小枚举 + 系统支持用户扩展（推荐）**
- **首版大类枚举**：`政府往来` / `会议文件` / `法律意见` / `签署件` / `财务报表` / `其他`
- **子类型**：用户可在大类下自定义；分类字典存独立 `Category` 集合，支持增删。
- **注意**：这是筛选 / 搜索的地基，必须早定骨架（首版可只上 6 个大类，子类型开放自定义）。

### D5 — 搜索范围：是否覆盖 PDF 内文
**✅ 定稿：首版只搜元数据，PDF 内文检索列为二期（推荐）**
- **理由**：PDF 内文需解析 + 索引（影响 R2 存储层与构建成本）。首版用 MongoDB `$text` 覆盖元数据（文件名 / 类型 / 公司 / 标签）即可满足"记得文件名"的诉求。
- **方案**：沿用 M2.1 已完成的 `$text` 索引 + 正则回退；PDF 内文检索待 R2 + 解析服务就绪后作为 M6 二期加。
