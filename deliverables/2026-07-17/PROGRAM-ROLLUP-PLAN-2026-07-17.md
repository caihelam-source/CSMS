## 2. 实施铺排（4 批次 + 2 门禁，依赖驱动）

> 排期原则：**地基先于体验、体验先于广度、开发可并行于运维门禁**。Wave 0/1/2 是"能不能用"，Wave 3/4 是"好不好用/多端可用"。

### 2.1 批次地图（含依赖与退出标准）

| 批次 | 主题 | 含里程碑 | 关键交付 | 硬依赖 | 退出标准（Exit Criteria） |
|---|---|---|---|---|---|
| **Wave 0** | 地基与安全门禁 | Phase 0 + R2 改造 | 修 live 权限、4 角色、seed-admin、关公开注册、R2 私有桶+预签名、/api/v1 | 用户拍板 E1–E5 | 真机真实登录后 `isAdmin/canEdit` 正确；R2 无公开 URL；公开注册已关；登录页可达 |
| **Wave 1** | 移动端硬化 + PWA | M3.6 | T-3.6.1–5 + manifest + service worker | Wave 0 权限修复 | 手机闭环成功率 100%；可安装到主屏；弱网可看已加载内容 |
| **Wave 2** | 业务闭环 | M5 + 2.0 模型 | T-M.1–4 + T-5.1–9（含确认弹窗/附件强控/状态机/归档审计/Dashboard 三来源） | Wave 0；D1/D2/D3 | 三来源闭环打通；签署/归档/解除均留 AuditLog；Dashboard 聚合正确 |
| **Wave 3** | 文件管理 | M6 | T-6.1–8（多级筛选+URL持久化/来源跳转/批量/空状态/只读/收藏/搜索并存） | Wave 2 命名/来源字段；D4/D5 | 找文件 ≤2 步；6 大类+可扩展；元数据搜索上线 |
| **Wave 4** | 微信 H5（路径 A） | M4 增量 | 微信网页授权+openid 绑定+JSSDK 分享+深链+移动公司切换器+预签名直传 | Wave 0/1/2；公众号资质 | 微信里点开即进、分享卡片可达、三端状态一致 |
| **Wave 5** | 小程序（路径 B，延后） | M4 | 独立小程序+BFF+wx.login→JWT | Wave 2 签署状态模型；小程序账号资质 | 小程序可签且状态与 Web 同步（**暂不排入期，待 Wave 4 验证后定**） |

### 2.2 运维门禁（Ops Gate，与开发并行，不卡开发）
| 门禁 | 内容 | 责任方 | 卡点 |
|---|---|---|---|
| **Gate A 数据迁移** | `mongodump` 备份 → `migrate-v5.js --apply`（生产库加 `--i-know-this-is-prod`） | 用户 + DBA（我提供脚本与复核清单） | 等 Atlas URI + DBA 复核 |
| **Gate B 真实后端** | Render 设 `VITE_USE_MOCK=false` + `VITE_API_BASE`；重建后 live 验证 | 用户（我出配置清单） | 环境变量 + 重建 |

> 关键判断：**Gate A/B 是"上线门禁"而非"开发阻塞"**。Wave 0–4 的开发完全可在 mock 下并行推进；Gate A/B 在**上线前**清掉即可。这样不浪费开发时间等 Atlas/DBA。

### 2.3 推荐执行顺序（含并行机会）
```
[Gate A 迁移] ─┐
[Gate B 真后端]─┴─(并行, 运维)──────────────────────────→ 上线门禁
                                                 
Wave 0 地基 ─→ Wave 1 移动端 ─→ Wave 2 闭环 ─→ Wave 3 文件 ─→ Wave 4 微信H5
   (可并行分析：PM 出 Wave4 PRD / 架构师出 Wave2 详细分解，不占工程师产能)
```
- Wave 0 必须先做（地基），但**其中的架构细化（高见远）可前置于编码**，让工程师一到位就有清晰 spec。
- Wave 1/2 是 P0，必须做完才能说"系统可用"；Wave 3/4 是 P1/增量，可紧随或按需裁剪。

## 3. 人员铺排（RACI + 产能）

> 本团队为虚拟协作团队（主理人齐活林协调，4 名专家角色由子智能体/专家承担）。现实约束：**编码产能集中在工程师寇豆码一人**，因此批次必须顺序占用其产能；而 PM/架构师的分析工作可前置并行，不与之争抢。

### 3.1 角色与在各批次的职责（RACI）
| 角色 | Wave 0 | Wave 1 | Wave 2 | Wave 3 | Wave 4 | Gate A/B |
|---|---|---|---|---|---|---|
| **许清楚（PM）** | C（确认验收口径） | A（签验收） | A（签验收） | A（签验收） | R（出增量 PRD） | I |
| **高见远（架构师）** | R（细化 spec：预签名/权限矩阵/v1 映射） | C（评审响应式方案） | R（M5 详细任务分解+服务层设计） | C（评审） | R（微信授权/BFF spec） | I |
| **寇豆码（工程师）** | R（全量实现） | R（全量实现） | R（全量实现） | R（全量实现） | R（全量实现） | — |
| **严过关（QA）** | R（安全回归：权限/预签名/注册关闭） | R（回归+移动端用例） | R（闭环逻辑+状态机用例） | R（筛选/搜索用例） | R（多端一致性用例） | — |
| **齐活林（交付总监）** | A（总控+用户拍板） | A | A | A | A | A（协调用户/DBA） |

图例：R=执行 / A=负责拍板与验收 / C=被咨询 / I=知会

### 3.2 产能与节奏提示
- **瓶颈**：唯一工程师。若用户希望加速，有两种解法——① 把 Wave 3（P1）与 Wave 4 部分 UI 工作后置，先交付 P0（Wave 0/1/2）；② 未来若工作量超出单工程师产能，再评估引入第二名工程师（需拆分独立模块避免耦合，如文件管理独立一人）。
- **并行红利**：Wave 0 编码期间，许清楚可并行起草 Wave 4 的微信增量 PRD、高见远可并行产出 Wave 2 的详细任务分解，均不占工程师时间。
- **QA 卡点**：Wave 0（安全）与 Wave 2（闭环逻辑）是高风险批次，QA 必须全量回归；Wave 1/3 可常规回归。

## 4. 达到的效果（统一效果指标）

> 把 UX 研究的量化指标、架构的安全目标、业务闭环目标**合并为一套前后对比**，作为每个批次的验收基线。

### 4.1 体验与业务效果（前后对比）
| 维度 | 当前（2026-07-17） | 完成后目标 | 由哪个批次达成 |
|---|---|---|---|
| 手机完成「看纪要→建任务→上传→归档」 | ≈0%（遮挡阻断） | **100%** | Wave 1 + Wave 2 |
| 真实后端登录后权限 | 全 false（游客，上线即崩） | **isAdmin/canEdit/canDelete 正确** | Wave 0 |
| 自动建 Task 准确率 | 无 | **≥95%**（硬动作识别+人工确认） | Wave 2 |
| 找目标文件平均步数 | ~5 步 | **≤2 步** | Wave 3 |
| SUS 可用性评分 | 未测 | **≥80（良好）** | Wave 1–3 综合 |
| 归档误操作率 | — | **0**（确认框+副本策略+审计） | Wave 2 |
| 多端状态一致（Web/手机/微信） | 无 | **三端共享同一 SignTask/Document 真相源** | Wave 0/2/4 |

### 4.2 安全与合规效果（这是 Wave 0 的"隐藏大单"）
| 风险（当前） | 完成后 |
|---|---|
| R2 返回公开 URL，签字件谁拿链接都能下（合规红线） | **私有桶 + 短期预签名 URL**，无公开地址 |
| 公开 register 可自封 admin（越权） | **关公开注册 + invite-token**，新用户走邀请 |
| 角色词汇栈内矛盾、无服务端执行 | **4 角色统一 + 服务端权限矩阵** |
| 演示自动登录掩盖真实登录页 | **显式开关 + 角标**，生产绝不默认自动登 |
| JWT 7d 无吊销 | **refresh token + logout 吊销 + isActive 即时拒** |

### 4.3 业务价值（给用户本人的）
- **合规动作真正闭环留痕**：会议→签署→归档在系统内一气呵成，未来审计可一键还原（直击 Vincent"合规守门人"核心动机）。
- **在路上也能办公**：手机/微信点开即用、弱网看已加载纪要、跨端接着签——匹配常飞 HK/沪/杭的工作方式。
- **3 家公司一套系统切换**：集团视图 + 行级过滤，CFO 跨司、秘书本司，零重复建模。
