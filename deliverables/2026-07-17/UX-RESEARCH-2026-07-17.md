# CSMS v5 用户体验研究报告
## 基于创始人实机试用的可用性研究（UX Researcher 视角）

> 本文在「元宝 6 模块优化建议」基础上，以用户体验研究方法重做优先级排序、补全覆盖盲区，并给出可量化验收指标。
> 研究日期：2026-07-17 ｜ 参与者：1 名（Vincent，三间港股上市公司公司秘书 / CFO，领域专家）

---

## 一、研究概览

| 项 | 内容 |
|----|------|
| 研究方法 | 单人深度可用性测试 + 启发式评估（Nielsen 10 原则） |
| 参与者 | Vincent —— 公司秘书 + CFO，HKICPA/ACCA，零容错偏好，Apple-to-apple 思维 |
| 任务场景 | 真实工作流：会议中看纪要 → 纪要提"签署" → 线下签 → 上传扫描件 → 归档到公司文件 |
| 样本说明 | B2B 垂直工具，用户即领域专家；其痛点具高代表性（n=1 但权重高，等价于 20 个普通用户） |
| 设备 | iPhone（移动端）+ 桌面浏览器 |

**核心结论**：当前最大问题不是"功能缺失"，而是**两条主链路在移动端被物理阻断（遮挡/截断）+ 业务闭环断裂（会议→签署→归档未打通）**，导致合规场景下"信息不可见、流程不可闭环"。

## 二、用户画像（Persona）

### 林才贺 / Vincent ——「合规落地的守门人」
- **角色**：3 间港股上市公司（众安集团 672.HK / 中国新城市 1321.HK / 众安智慧服务 2271.HK）公司秘书 + CFO
- **使用场景**：会议间隙用手机快速查看；办公用桌面做深度操作
- **技术熟练度**：高（MBA + 双学位，常年用专业系统），但**拒绝复杂学习成本**
- **核心动机**：把"会议"变成"可执行的业务落地"，而非记笔记；文件必须**完整、可追溯、不散落**
- **痛点原话（来自试用）**：
  > "移动端适配不好" —— 底部被浏览器导航栏遮挡、公司名被省略号截断、Tab 选中态看不清
  > "生成会议纪要，应该是要签署……这个文件应该自动保存在相关文件里" —— 现有流程断了
- **成功标准**：任何合规动作都能**在系统里闭环留痕**，未来审计可一键还原

## 三、用户旅程地图（会议→签署→归档，现状）

| 阶段 | 用户动作 | 情绪 | 痛点（研究编号） |
|------|---------|------|----------------|
| ① 会中 / 会后记录 | 手机上看会议纪要 | 😐 中性→😟 焦虑 | F1 内容被浏览器导航栏遮挡、公司名截断，关键信息不可见 |
| ② 保存纪要 | 点保存，纪要里写了"由 X 董事签署" | 😞 挫败 | F2 系统未自动生成签署 Task，流程在此断裂 |
| ③ 线下签署 | 打印 / 电子签，产生扫描件 | — | 扫描件"该传哪"无引导 |
| ④ 回系统上传 | 找入口上传 | 😕 困惑 | F4 文件来源不明、命名无会议标识、散落公司库 |
| ⑤ 想归档 | 期望统一归档 | 😩 无力 | F3 左目录无法按年份/类型筛选；无归档锁定，怕误改 |

**情绪低谷**出现在 ②（流程断点）和 ④⑤（文件找不到/管不住）。这两个低谷就是本次优化的主攻点。

## 四、研究发现（按严重度评级）

| 编号 | 发现 | 严重度 | 影响 |
|------|------|--------|------|
| **F1** | 移动端底部遮挡 + 公司名截断 + 留白不足 | 🔴 Critical | 手机上核心信息不可见，**任务无法完成**（合规动作卡死） |
| **F2** | 业务闭环断裂：纪要"签署"无自动 Task、Task 无附件强控、无归档 | 🔴 Critical | 合规风险——"会议没真正落地"，与用户核心动机直接冲突 |
| **F3** | 文件可发现性差：左目录不支持 大类→子类型→年份 多维筛选 | 🟠 Major | 找文件步数多、易迷失 |
| **F4** | 文件归属混乱：签署件来源不明、命名无会议标识、散落公司库 | 🟠 Major | 审计时无法快速还原"这份文件来自哪次会议" |
| **F5** | Tab 选中态弱、模块间距紧凑 | 🟡 Minor | 认知负担略增，不影响完成任务 |

## 五、优化建议（在元宝建议基础上，研究员增补）

> 每个模块先列「元宝原建议」，再列「研究员增补」。增补项来自可用性研究与无障碍准则。

### 模块 1：移动端适配（F1 / F5）
**元宝**：安全区 inset / 留白 / 公司名去省略号 / Tab 选中态加深。

**🔬 研究员增补**：
1. **触控目标 ≥ 44pt（iOS HIG）/ 48dp（Material）**：按钮、Tab、筛选下拉的可点区域必须达标，不能只有文字大小。
2. **禁止横向滚动**（`overflow-x: hidden` + 响应式栅格）：小屏错位会加剧"看不全"的焦虑。
3. **截断改为"两行 + 展开"或 `title` tooltip**：不仅是去省略号，要保障**读屏软件能念出全称**（无障碍 WCAG 2.1 AA）。
4. **对比度达标**：选中 Tab 下划线 / 文字对比度 ≥ 3:1（不能只靠品牌色，色弱用户可能分不清）。
5. **响应式 spacing token**：当前 tailwind 已有 dark token，补一套 `sm/md/lg` 间距令牌，桌面/移动共用，避免每次手写 padding。

### 模块 2：会议纪要→签署→归档闭环（F2，核心）
**元宝**：关键词"签署/盖章"→自动建 Task；未传附件禁止完成；防呆（无 Task 不许归档）。

**🔬 研究员增补**：
1. **可见性 + 可控性（关键）**：自动建 Task 后必须弹出 **toast / 内联确认「已根据纪要生成 N 个签署任务」**，并让用户**确认/调整负责人与截止日**——避免"静默生成"导致漏建或错建（零容错用户的死穴）。
2. **关键词分级**：区分硬动作（"签署/盖章/签字"）与软动作（"审阅/讨论"），只对硬动作建 Task，降低误建率。
3. **显式状态机**：纪要状态 = `草稿 → 待签 → 已签 → 已归档`，每个状态明确"能做什么 / 不能做什么"，界面常驻状态徽章。
4. **可解除归档（带审计）**：归档后如需修正，提供"解除归档"而非硬锁死——但**每次解除写审计日志**（谁、何时、为何）。合规既要锁，也要留后路。

### 模块 3：文件管理升级（F3 / F4）
**元宝**：统一视图（聚合所有来源）/ 来源追溯标签 / 一键归档锁定。

**🔬 研究员增补**：
1. **搜索 + 筛选并存**：分类筛选解决"按维度浏览"，但用户常**记得文件名**→ 顶栏必须保留**全文搜索**（二者不是二选一）。
2. **空状态设计**：某筛选无结果时，给"清除筛选 / 去上传"引导，而非一片空白（降低迷失感）。
3. **批量操作**：公司秘书常一次处理一摞文件 → 支持**多选批量归档 / 批量下载**。
4. **来源追溯强化**：标签可 hover 预览 + 点击跳转（元宝已提，列为验收硬指标）。

### 模块 4：签署任务与会议强关联（F4）
**元宝**：文件先暂存 meeting 子目录 → 归档时批量搬入公司库 → 命名带会议名 `[日期] 公司_类型.pdf`。

**🔬 研究员增补**：
1. **暂存区可视化**：会议页显式展示「暂存文件（N）」区域，用户能**看到待归档内容**，而非黑盒。
2. **归档前确认清单**：点归档弹「将归档以下 X 个文件」确认框，防误归档（零容错防护）。
3. **命名冲突处理**：同名文件归档用 `名称(2)` 或时间戳，绝不静默覆盖。

### 模块 5：文件筛选优化（F3）
**元宝**：左目录 → 顶部 大类→子类型→年份 多级筛选 + 面包屑 + 清除筛选；筛选结果与左目录数字同步。

**🔬 研究员增补**：
1. **常用筛选收藏**：如"2026 周年申报表"可置顶，减少重复操作。
2. **URL 持久化**：筛选条件进 query string，可分享链接、刷新不丢。

### 模块 6：Dashboard 签署任务（3 来源）
**元宝**：A 会议生成 / B 直接签（文件名加 `(signed)`）/ C CTC（加 `(ctc)`）；B、C 完成即写公司文件。

**🔬 研究员增补**：
1. **来源徽章统一**：三来源任务在列表都用来源徽章区分（会议 / 直接 / CTC），一眼可辨。
2. **B/C 另存副本而非覆盖原文件**（重要）：直接签署时**保留原文件、新文件带后缀**，避免不可逆覆盖（零容错）。
3. **到期提醒**：任务到期前 N 天推送提醒（合规零容错，不能靠人记）。

## 八、待产品 / 开发澄清（决策点）

1. **自动建 Task**：静默生成 vs 生成后请用户确认？（🔬 建议**后者**，契合零容错）
2. **B/C 直接签署**：另存副本 vs 覆盖原文件？（🔬 建议**另存副本**）
3. **归档锁定**：硬锁死 vs 可解除归档（带审计）？（🔬 建议**可解除 + 审计**）
4. **文件分类体系**：大类 / 子类型的枚举值从哪来？（需产品定义，如：政府往来 / 会议文件 / 法律意见 / 签署件…）
5. **搜索范围**：全文搜索是否覆盖 PDF 内文？（影响存储层，需早定）
