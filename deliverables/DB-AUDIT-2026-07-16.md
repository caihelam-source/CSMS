# CSMS 中央数据库架构审核报告

> 审核人：数据库优化专家（MongoDB 视角）
> 审核日期：2026-07-16
> 审核范围：12 个 Mongoose 模型 + 关键路由 + 360° 聚合管道 + 全局搜索 + 连接配置
> 技术栈：Node/Express + MongoDB (Mongoose) + MongoDB Atlas

---

## TL;DR — 最该先动的三件事

1. **360° 聚合里 `$lookup` 滥用 `$expr`，把 `Company.links` 上的索引废了。**
   公司集合越大越慢（每次请求全表扫 companies）。改成 `localField/foreignField` 直接 lookup，索引立刻生效。
2. **Dashboard 多个 `countDocuments` 没有索引支撑。**
   `ComplianceReminder`、`Task`、`Company.status` 的过滤字段全无索引，每次进首页 = 全表扫。
3. **列表接口无分页、无投影。**
   `GET /companies`、`GET /personnel` 一上来 `find().sort()` 返回**整篇文档全量**，数据一多内存/带宽就爆。

## P0 — 性能与正确性（立刻改）

### 1. `$expr` 在 `$lookup` 子管道中废掉索引 ⭐⭐⭐

**位置**：`server/routes/personnel.js` → `getByPersonnelAPI`（第 113–232 行）

**问题**：5 个 `$lookup` 全部用 `$expr: { $in: [...] }` 做关联。MongoDB 在 `$lookup` 子管道里一旦碰到 `$expr`，**就不会使用 `foreign` 集合上的任何索引**。意味着每次查一个人的 360°，都要对 `companies` / `tasks` / `meetings` / `compliancereminders` 做全集合扫描 + 内存过滤。

特别是 companies 那段（第 117–156 行）：
```js
$lookup: {
  from: 'companies',
  let: { pid: '$_id' },
  pipeline: [
    { $match: { $expr: { $in: ['$$pid', '$links.link'] } } },  // ❌ 索引失效
    ...
  ],
  as: 'companies'
}
```
而你在 `Company` 上明明建了 `links.link` 和 `{links.linkModel, links.link}` 索引（Company.js 第 80–83 行）——全白建了。

**修复方案（最高性价比）**：
- **companies** 改用经典 `localField/foreignField` lookup（**确定能用上 `links.link` 索引**），再用 `$map`+`$filter` 只保留 `linkModel==='Personnel' && link===pid` 的 link。

  ```js
  {
    $lookup: {
      from: 'companies',
      localField: '_id',
      foreignField: 'links.link',   // ✅ 使用 links.link 索引
      as: 'companyDocs'
    }
  },
  {
    $addFields: {
      companies: {
        $map: {
          input: '$companyDocs',
          as: 'c',
          in: {
            $let: {
              vars: {
                ml: {
                  $filter: {
                    input: '$$c.links', as: 'l',
                    cond: { $and: [
                      { $eq: ['$$l.linkModel', 'Personnel'] },
                      { $eq: ['$$l.link', '$_id'] }
                    ] }
                  }
                }
              },
              in: {
                company: { _id: '$$c._id', name: '$$c.name', nameChinese: '$$c.nameChinese',
                           registrationNumber: '$$c.registrationNumber', type: '$$c.type', status: '$$c.status' },
                roles: '$$ml.roles', shares: '$$ml.shares', shareType: '$$ml.shareType',
                appointmentDate: '$$ml.appointmentDate', cessationDate: '$$ml.cessationDate', notes: '$$ml.notes'
              }
            }
          }
        }
      }
    }
  }
  ```

- **tasks / meetings / reminders**：这几个需要 "公司列表 IN" 关联，`$lookup` 子管道里用 `$in` 对变量数组**必然**走 `$expr`、无法用索引。最干净的做法是**移到应用层并行查询**——`Model.find({ company: { $in: cids } })` 这种**顶层 `$in` 是能命中 `company` 索引的**：
  ```js
  const companyIds = companies.map(c => c.company._id);
  const [tasks, meetings, reminders, documents] = await Promise.all([
    Task.find({ $or: [{ company: { $in: companyIds } }, { personnel: personId }] }).sort({ dueDate: 1 }),
    Meeting.find({ $or: [{ company: { $in: companyIds } }, { 'attendees.ref': personId }] }).sort({ scheduledAt: -1 }),
    ComplianceReminder.find({ company: { $in: companyIds } }).sort({ dueDate: 1 }),
    Document.find({ personnel: personId }).sort({ createdAt: -1 }),
  ]);
  ```
  这样既保持单次 HTTP 响应、又让每个集合都走索引。

### 2. Dashboard 统计缺索引 ⭐⭐⭐

**位置**：`server/routes/companies.js` → `/stats/dashboard`（第 63–95 行）

**问题**：12 个 `countDocuments` 里，这些**没有任何索引**支撑：
- `ComplianceReminder.countDocuments({ status: {$in:[...]}, dueDate: {$gte:now} })` → 无索引，全表扫
- `Task.countDocuments({ status: 'pending' })` → 只有单列 `status` 索引，但配合无排序还好；但 `upcomingReminders/expiredReminders` 是 `status + dueDate` 复合过滤，建议复合索引
- `Company.countDocuments({ status: 'active' })` → `status` 无索引

**修复**：在对应模型补复合索引：
```js
// ComplianceReminder.js
complianceReminderSchema.index({ status: 1, dueDate: 1 });        // dashboard 过滤
complianceReminderSchema.index({ company: 1, dueDate: 1 });       // 公司维度提醒列表

// Task.js
taskSchema.index({ status: 1, dueDate: 1 });                     // pending/overdue 统计
taskSchema.index({ company: 1, status: 1 });                    // 公司任务看板

// Company.js
companySchema.index({ status: 1 });                              // active 统计 / 状态筛选
```

> 注意：MongoDB 复合索引字段顺序遵循 ESR 规则（等值 → 排序/范围）。`status`（等值）+ `dueDate`（范围）顺序正确。

### 3. 列表接口无分页、无投影 ⭐⭐⭐

**位置**：
- `companies.js` 第 34 行 `Company.find(query).sort({ name: 1 })` — 返回**整篇公司文档**（含可能很长的 `links[]`）
- `personnel.js` 第 46 行 `Personnel.find(query).sort({ name: 1 })` — 无 limit

**问题**：没有 `limit/skip` 或游标分页，也没有 `.select()` 投影。公司一多（links 数组膨胀后），单次响应可能几 MB，且 `Person.find` 还附带两次 aggregate 派生 roles（见 P1-D）。

**修复**：加分页 + 投影。建议封装一个统一分页 helper：
```js
function paginate(query, opts = {}) {
  const page = Math.max(1, parseInt(opts.page) || 1);
  const limit = Math.min(parseInt(opts.limit) || 25, 100);
  return query.skip((page - 1) * limit).limit(limit);
}
// 使用
const companies = await Company.find(query)
  .select('name nameChinese registrationNumber type status links.link links.roles links.linkModel')
  .sort({ name: 1 })
  .skip((page-1)*limit).limit(limit);
const total = await Company.countDocuments(query);
```
前端列表用 `total` 做分页器。游标分页（基于 `_id` 或 `createdAt`）比 `skip` 在大数据量下更高效，但若当前规模 < 10k 文档，`skip/limit` 足够。

## P1 — 数据一致性与可维护性

### 4. 遗留集合仍被双写 → 事实源漂移 ⭐⭐⭐

**位置**：`personnel.js` `merge` 路由（第 421–430 行）；`index.js` 第 57–58 行仍挂载 `companyEntriesRoutes`/`companyRegisterRoutes`

**问题**：v5.0 已把董事/股东统一进 `Company.links`（单一事实源）。但 `merge` 时**仍然同时更新** `ShareholderEntry` 和 `DirectorEntry`：
```js
await ShareholderEntry.updateMany({ personnelRef: sourceId }, { $set: { personnelRef: targetId } });
await DirectorEntry.updateMany({ personnelRef: sourceId }, { $set: { personnelRef: targetId } });
```
迁移脚本 `--apply` 后这两张遗留表本应废弃，现在还在写 → 双事实源，迟早漂移。且 `index.js` 仍暴露 `/companies/:id/...` 的 shareholder/director entries 路由。

**修复**：迁移完成确认后——
- 在 `merge` 路由里把这两段 `updateMany` 用 `if (process.env.KEEP_LEGACY_ENTRIES)` 之类的开关包起来（默认关）。
- 把 `companyEntriesRoutes` 标记为只读 / 或直接废弃，前端不再写入，只读视图从 `Company.links` 派生。
- 数据确认无误后 `db.shareholderentries.drop()` / `db.directorentries.drop()`。

### 5. `Personnel.roles`：存储值 vs 读时派生混用 → 不可信 ⭐⭐

**位置**：`Personnel.js` 第 14 行 `roles: [{type:String}]`；`personnel.js` 第 408 行 merge 时更新 stored roles；但普通 link 编辑（companies.js 第 269–351 行）**只写 `Company.links`，不回写 `Personnel.roles`**。

**问题**：`Personnel.roles` 在 merge 时更新、在 link 增删时不更新，二者必然不一致。任何直接读 `personnel.roles` 的地方都会拿到陈旧数据（前端 PersonnelDetail 实际用的是读时派生，所以目前"侥幸"正确，但这是个定时炸弹）。

**修复**：二选一，**不要混**：
- **方案 A（推荐）**：彻底删除 `Personnel.roles` 字段，所有角色一律从 `Company.links` 读时派生。干净、无漂移。
- **方案 B**：保留 stored roles，但在 `companies` 的 link CRUD（POST/PUT/DELETE links）里同步维护 `Personnel.roles`（用 arrayFilters 增删角色）。代价是每次 link 变更多一次 Personnel 写。

### 6. 删除实体不清理反向引用 → 悬空指针 ⭐⭐

**位置**：
- `companies.js` 第 173–182 行 `DELETE /companies/:id` → 直接 `findByIdAndDelete`，但 `Document/Meeting/Task/ComplianceReminder/SignTask` 里 `company` 字段仍指向它。
- `personnel.js` 第 315–323 行 `DELETE /personnel/:id` → 直接删人，但 `Company.links[].link`、`Meeting.attendees[].ref`、`Document.personnel`、`SignTask.signers[].signer` 全部悬空。360° 聚合里 `$lookup` 会返回指向已删 `_id` 的 company doc，`reverse-links` 也会出错。

**修复**：删除前清理引用，或改用软删除（`isDeleted`/`deletedAt` 标志，查询默认过滤）。示例（删公司）：
```js
await Company.findByIdAndDelete(id);
await Promise.all([
  Document.updateMany({ company: id }, { $unset: { company: '' } }),
  Meeting.updateMany({ company: id }, { $unset: { company: '' } }),
  Task.updateMany({ company: id }, { $unset: { company: '' } }),
  ComplianceReminder.deleteMany({ company: id }),
  SignTask.updateMany({ company: id }, { $unset: { company: '' } }),
  Company.updateMany({ 'links.link': id }, { $pull: { links: { link: id } } }),
]);
```

### 7. `companies.js` 重复路由定义（Bug）⭐⭐

**位置**：`companies.js` 第 108–128 行 **和** 第 130–150 行——`GET /reverse-links/:personnelId` 被定义了**两次**，内容完全相同。Express 后者静默覆盖前者（行为一致但纯属隐患），而且与第 42–60 行的 `GET /reverse-links?personnelId=` 形成**两套不一致接口**。

**修复**：删掉重复的一份，并把 query-param 版本统一为 path-param 版本（前端统一调用 `/reverse-links/:personnelId`）。

### 8. `Document.generateDocNumber` 竞态 + 全表扫 ⭐⭐

**位置**：`Document.js` 第 41–48 行
```js
const count = await this.countDocuments({ type: type || 'other' });
const seq = String(count + 1).padStart(4, '0');
return `${prefix}-${year}-${seq}`;
```
**问题**：并发创建同类型文档时，`count+1` 会算出相同序号（竞态）；且每次都全表 `countDocuments`。

**修复**：用 `counters` 集合做原子自增（`findOneAndUpdate({_id:'DOC-'+type},{$inc:{seq:1}},{upsert:true,returnDocument:'after'})`），或基于 `_id`/时间戳生成唯一号。若保留当前格式，至少给 `docNumber` 加 `unique` 索引 + 失败时重试。

## P2 — 可扩展性 / 效率

### 9. 全局搜索用正则而非 `$text` 索引 ⭐⭐

**位置**：`search.js` 第 102–104 行。6 个实体各跑一次 `$or` 正则（大小写不敏感）。`Personnel` 上明明有 `text` 索引（Personnel.js 第 30 行），但没用上。

**问题**：大小写不敏感正则 = **全集合扫描**，且无法按相关度排序。6 个实体并行也扛不住大数据量。

**修复**：
- 各实体加 `$text` 索引（Personnel 已有；Company/Document/Meeting/Task/ComplianceReminder 补），搜索改用 `$text: { $search: q }` + `$meta: 'textScore'` 排序。
- 或在 Atlas 上启用 **Atlas Search**（支持中文分词、模糊、高亮），一步到位。

### 10. Mongoose 连接无连接池/超时配置 ⭐⭐

**位置**：`index.js` 第 67 行 `mongoose.connect(MONGO_URI)` —— 没有任何选项。

**修复**：在 Render 多实例下显式配置池：
```js
mongoose.connect(MONGO_URI, {
  maxPoolSize: 10,            // 按实例规格调
  minPoolSize: 2,
  socketTimeoutMS: 30000,
  serverSelectionTimeoutMS: 5000,
  retryWrites: true,
});
```
生产环境建议 `autoIndex: false`（索引由迁移脚本管理，避免每次冷启动建索引拖慢部署）。

### 11. Excel 导入写入了 schema 不存在/类型不符的字段 ⭐⭐

**位置**：`companies.js` 第 225–232 行 `import/excel`：
```js
region: String(row['地区'] || '').trim(),              // ❌ Company schema 无 region 字段（只有 jurisdiction），会被 strict 模式丢弃
financialYearEnd: String(row['财务年度结束'] || '').trim(),  // ❌ schema 期望 {day,month}，这里写字符串
```
**问题**：`region` 静默丢失；`financialYearEnd` 类型不匹配（应为数字子文档），要么被丢弃要么存成脏数据。

**修复**：映射改为 `jurisdiction`（已有字段）；`financialYearEnd` 解析为 `{ day, month }` 或先补 `financialYearEnd` 子文档支持字符串格式。

### 12. `Document.expiresAt` 缺索引 ⭐

前端有"文件到期"徽章（docExpiryStatus），到期查询 `expiresAt < now` 无索引。补：
```js
documentSchema.index({ expiresAt: 1 });
documentSchema.index({ company: 1, expiresAt: 1 });   // 公司维度到期清单
```

### 13. `SignTask.signers[].signer` 缺索引 ⭐

若未来要"查某人所有待签任务"，需 `SignTask` 上 `signers.signer` 索引。当前规模可暂缓。
